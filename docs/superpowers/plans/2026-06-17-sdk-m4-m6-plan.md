## M4 — Jira publish + AI draft + review UI

### Backend
1. **`POST /reports/:id/publish`** (matched before `/:asset`): build on `handleJiraIssue`'s skeleton; **explicit `{stubbed}` response** when unconfigured; reporter→ADF body + validated `qa-reporter:<email>` label; `projectKey` from request, **allowlist-validated**; reuse `adf.ts` + absolute URL. **Origin-allowlisted** (F4-AUTH).
2. **`GET /jira/epics?q=`** server-side: input allowlist (charset + max length); JQL = `project = <allowlisted projectKey> AND summary ~ <escaped q> AND issuetype = Epic` (**project forced server-side**); **Origin-allowlisted**; stub when unconfigured. Security test: malicious `q` cannot widen scope.
3. **AI-polish = reuse `POST /jira/draft`** (F4-AI). No new AI route.

### SDK
4. **Control-flow inversion (iss-4, corrected by gate round 1):** in the jira flow (`jira.enabled` **and** `endpoint`), `stop()` **uploads the bundle immediately** (so a `reportId` exists for AI-polish), then opens the review drawer; **onSubmit/download are suppressed** while it is open. **Publish** → `/jira/draft` (if AI) → `/reports/:id/publish` → post-publish view; **Cancel** → close, no publish. The **non-jira flow is unchanged** (eager `onStop`/`onSubmit`/download — the existing `Bugzar.test.tsx` contract; the inversion branches on `jira.enabled`, so that suite stays green). **No `endpoint`/`jira` → no drawer at all** (callbacks-first; there is no in-drawer edit surface, so no data-loss claim). *(iss-7/8/9 — drawer state machine, keyboard/focus, upload-failure UX, and final nested prop shape — judged at M4's code gate.)*
5. **Review drawer (bounded):** in = title(required), description, AI-polish (gated; hidden when unavailable), AI-fail fallback, epic combobox (server search, debounced, ARIA, **prompt-to-search + `defaultEpicKey` pre-fill**), publish, post-publish view. out = onboarding/mode-pick/two-leg/save-for-later/resume/abandon. **English copy + `labels` prop.** One consistent FAB-suppression model with the picker; anchors to `position`.
6. **F4.3 (design-mode publish) — drawer reuse, NO backend change.** On a design pick (when `jira.enabled` + `endpoint`), upload the picked annotations as the **`design` asset** (`design.json` — the backend `ASSETS` slot that `handleDesignDraft` already reads; **metadata-only**, no screenshots — the `/elements/:id.png` route is extension-only), then open the **review drawer in `mode:'design'`**: title/desc/epic/**AI-polish (`POST /jira/draft` with `mode:'design'` → `generateDesignDraft`)**/publish; body = **imageless element cards** (selector + note). `onAnnotate` still fires (no data loss); `onPublished` on publish; no `jira`/`endpoint` → unchanged (`onAnnotate`/download). SDK adds an `uploadDesign` path (**mapping each annotation `note`→`userNote`** — the field `normalizeDesignElements` reads; `DesignAnnotation` uses `note` but the backend's element shape uses `userNote`, so reviewer notes survive into the AI draft + stub) + `assetUrls.design` type; the issue stays a Bug carrying the design ADF body (`/reports/:id/publish` is mode-agnostic). Tests: `design-publish.test.tsx` (design-asset upload, drawer open + imageless cards, draft `mode:'design'`, publish→onPublished, no-jira guard).

**Gate:** backend+sdk tests green · e2e (demo→record→review→stub-publish, `stubbed` honesty, absolute URL) · M4 code gate.
